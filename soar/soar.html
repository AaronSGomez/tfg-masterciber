Laboratorio SOAR

Guía de despliegue con bloques de código copiable

Este manual visual y operativo detalla el despliegue del entorno SOAR (Shuffle) en el laboratorio. Incluye los requisitos de mapas de memoria para Elasticsearch, el descriptor docker-compose.yml para levantar el orquestador, y las configuraciones de automatización y verificación de logs listas para copiar.

1. Requisito previo crítico

Antes de levantar los contenedores, el host debe permitir suficientes mapas de memoria virtual para Elasticsearch. Ejecuta el comando siguiente como root en la máquina anfitriona.

Host Linux 
sysctl -w vm.max_map_count=262144

Para hacerlo permanente, añade vm.max_map_count=262144 al final de /etc/sysctl.conf.

2. Archivos que debes crear

En el mismo directorio del docker-compose.yml deben existir estos tres archivos:

  • docker-compose.yml
  • thehive.conf
  • cortex.conf

3. Arranque inicial de la infraestructura de datos

Primero se levantan solo Cassandra y Elasticsearch. Después se monitoriza el log de Cassandra hasta que termine de inicializarse.

Arranque inicial 
docker compose up -d cassandra elasticsearch
docker logs -f thehive_cassandra

No avances hasta ver un mensaje equivalente a Created default superuser role 'cassandra' o Startup complete.

4. Cortex: configuración inicial y creación de la organización

Cuando los motores de datos estén estables, levanta Cortex y completa la configuración inicial en la interfaz web.

Arranque de Cortex 
docker compose up -d cortex
  1. Abre http://localhost:9001.
  2. Inicia sesión con admin y secret.
  3. Crea la organización soar_lab.
  4. Crea el usuario de API y asígnale la organización y los roles necesarios.
  5. Genera la API key y cópiala inmediatamente.

5. Vinculación en TheHive y ajuste de la API key

6. Construcción del workflow de prueba en n8n

Crea un flujo nuevo en n8n y enlaza Webhook, TheHive y Cortex con la configuración exacta del laboratorio.

1. Creación del disparador Webhook

n8n
  • En el menú izquierdo de n8n, entra en Workflows y pulsa Create a workflow.
  • Elimina el nodo manual por defecto When clicking Execute Workflow.
  • Haz clic en + y añade el nodo Webhook.
  • Configura Authentication: None, HTTP Method: POST y Path: wazuh-alerts.
  • Copia la URL de producción generada por n8n en formato http://<IP_DE_N8N>:5678/webhook/wazuh-alerts.

2. Integración del nodo TheHive

TheHive
  • Haz clic en + a la derecha del Webhook y busca TheHive.
  • Selecciona la acción inicial Case → get many.
  • Pulsa Add Node para añadirlo al lienzo.
  • Configura Credential for TheHive v4 con la credencial API creada en el paso 1.
  • Define Resource: Case y Operation: Get Many / Get All.
  • En Settings, activa Always Output Data para que el flujo continúe aunque no haya casos iniciales.

3. Integración del nodo Cortex

Cortex
  • Haz clic en el + a la derecha del nodo de TheHive y busca Cortex.
  • Elige la acción inicial Job → get.
  • Añádelo con el botón Add Node.
  • Configura Credential for Cortex con la credencial API de Cortex creada en el paso 2.
  • Escribe Job ID: 1 como parámetro de prueba inicial.
  • En Settings, activa Continue On Fail para evitar bloqueos cuando el Job 1 no exista en una base limpia.

4. Activación del flujo

Activo
  • En la esquina superior derecha de n8n, cambia el interruptor a Active.
  • El flujo queda guardado en producción y escuchará de forma permanente sin ejecuciones manuales.

7. Archivos completos de configuración

Aquí tienes los dos ficheros de configuración y el orquestador listos para copiar directamente.

docker-compose.yml 
version: '3.8'

services:
  elasticsearch:
    image: elasticsearch:7.17.9
    container_name: thehive_elasticsearch
    environment:
      - discovery.type=single-node
      - xpack.security.enabled=false
      - "ES_JAVA_OPTS=-Xms512m -Xmx512m"
    deploy:
      resources:
        limits:
          memory: 1024M
    volumes:
      - elasticsearch_data:/usr/share/elasticsearch/data
    networks:
      - security_net

  cassandra:
    image: cassandra:3.11
    container_name: thehive_cassandra
    environment:
      - MAX_HEAP_SIZE=512M
      - HEAP_NEWSIZE=128M
      - CASSANDRA_CLUSTER_NAME=TheHive
      - CASSANDRA_CONCURRENT_READS=8
      - CASSANDRA_CONCURRENT_WRITES=8
      - CASSANDRA_CONCURRENT_COMPACTORS=1
    deploy:
      resources:
        limits:
          memory: 1024M
    volumes:
      - cassandra_data:/var/lib/cassandra
    networks:
      - security_net

  cortex:
    image: thehiveproject/cortex:3.1.7
    container_name: cortex
    depends_on:
      - elasticsearch
    environment:
      - job_directory=/opt/cortex/jobs
    ports:
      - "9001:9001"
    deploy:
      resources:
        limits:
          memory: 750M
    volumes:
      - ./cortex.conf:/etc/cortex/application.conf
      - /var/run/docker.sock:/var/run/docker.sock
    networks:
      - security_net

  thehive:
    image: thehiveproject/thehive4:4.1.24
    container_name: thehive
    depends_on:
      - cassandra
      - elasticsearch
      - cortex
    ports:
      - "9000:9000"
    deploy:
      resources:
        limits:
          memory: 1024M
    volumes:
      - thehive_data:/etc/thehive
      - ./thehive.conf:/etc/thehive/application.conf 
    command: --config-file /etc/thehive/application.conf
    networks:
      - security_net

  n8n_db:
    image: postgres:16-alpine
    container_name: n8n_postgres
    environment:
      - POSTGRES_USER=n8n_user
      - POSTGRES_PASSWORD=n8n_secure_pass
      - POSTGRES_DB=n8n_database
    deploy:
      resources:
        limits:
          memory: 256M
    volumes:
      - n8n_db_data:/var/lib/postgresql/data
    networks:
      - security_net
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U n8n_user -d n8n_database"]
      interval: 5s
      timeout: 5s
      retries: 5

  n8n:
    image: n8nio/n8n:latest
    container_name: n8n_automation
    depends_on:
      n8n_db:
        condition: service_healthy
    ports:
      - "5678:5678"
    deploy:
      resources:
        limits:
          memory: 512M
    environment:
      - DB_TYPE=postgresdb
      - DB_POSTGRESDB_HOST=n8n_db
      - DB_POSTGRESDB_PORT=5432
      - DB_POSTGRESDB_DATABASE=n8n_database
      - DB_POSTGRESDB_USER=n8n_user
      - DB_POSTGRESDB_PASSWORD=n8n_secure_pass
      - GENERIC_TIMEZONE=Europe/Madrid
      - N8N_SECURE_COOKIE=false
    volumes:
      - n8n_data:/home/node/.n8n
    networks:
      - security_net

volumes:
  elasticsearch_data:
  cassandra_data:
  cortex_data:
  thehive_data:
  n8n_db_data:
  n8n_data:

networks:
  security_net:
    driver: bridge

8. Despliegue global del laboratorio

Una vez ajustada la API key, despliega el resto del ecosistema y espera a que TheHive termine de inicializar sus tablas e índices.

Despliegue final 
docker compose up -d
docker logs -f thehive

La señal de éxito es Listening for HTTP on /0.0.0.0:9000.

9. Verificación final

1. Accede a TheHive en http://localhost:9000.

2. Define la contraseña de administrador en el primer inicio.

3. Comprueba que el icono de Cortex aparece en verde.

4. Si quieres, accede también a n8n en http://localhost:5678.

10. Registro de Incidencias Técnicas Previas

Incidencias ⚠️

1. Problemas de recursos y estabilidad. La máquina virtual colapsaba al arrancar todos los contenedores de forma simultánea debido a la sobrecarga de consumo de RAM de los servicios basados en Java (Elasticsearch, Cassandra, TheHive y Cortex). Se solucionó ampliando la VM del host a 8GB RAM y 4 Núcleos, configurando deploy.resources.limits en el docker-compose.yml y realizando un arranque escalonado.

2. Bloqueo de seguridad de cookies en n8n. Error de cookies seguras que impedía el acceso por HTTP inseguro. Corregido inyectando la variable de entorno N8N_SECURE_COOKIE=false en el servicio correspondiente.

3. Fallo de integración TheHive-Cortex (401 Unauthorized). Discordancia en el nombre de la organización entre los dos extremos y sintaxis obsoleta de entorno. Solventado implementando una configuración robusta vía archivo application.conf (HOCON) y alineando el nombre exacto de la organización en ambos sistemas.

4. Invisibilidad de Cortex en la Interfaz de TheHive. El operador no podía visualizar los analizadores de Cortex en TheHive debido al estado no org del usuario por defecto de administración global. Se solucionó creando una organización formal de trabajo (`Thehive`) y asignándole un usuario operador con el perfil de `org-admin` para realizar las consultas.
