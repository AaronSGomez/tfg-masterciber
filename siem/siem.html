Detonación del Exploit React2Shell RCE (CVE-2025-55182)

La vulnerabilidad crítica React2Shell está ligada a la deserialización insegura de componentes de servidor React (RSC Deserialization) en Next.js. El ataque se ejecutó remotamente desde una máquina externa utilizando la dirección IP móvil probada 178.139.228.237, apuntando al servidor físico Raspberry Pi 5 configurado con HTTPS seguro (puerto 443), haciendo uso del proxy inverso Apache2 y certificados firmados por Let's Encrypt.

Flujo Técnico del Ataque Identificado

  • 1. Interceptación: Se intercepta la petición inicial mediante Burp Proxy, identificando que el endpoint vulnerable /api/deserialize procesa un JSON en el que el campo rsc_payload en Base64 contiene la lógica de deserialización.
  • 2. Envío al Repeater: La solicitud se traslada a Burp Repeater. Se inyecta la payload maliciosa que hace uso de la función dinámica eval() nativa del backend para forzar la ejecución remota de comandos en el host Next.js.
  • 3. Ejecución y Reconocimiento: Se inyecta el comando ls -la /app. El servidor procesa el exploit y devuelve el árbol de directorios de desarrollo crítico (node_modules, .next, Dockerfile, package.json).
  • 4. Exfiltración de Bandera: Se inyecta el comando cat /app/flag.txt para leer la bandera de integridad introducida en el contenedor ciber_tfg-modern_app-1, extrayendo exitosamente la firma del alumno ||| AARONSGOMEZ — REMOTE SHELL ACCESS ||| ... sobre HTTPS en el puerto 443.

1. Caso de Uso SIEM y Alerta Disparada (Evidencia #18)

Dado que el proxy inverso de producción (el servidor web Apache2) reside físicamente en el sistema operativo host (la Raspberry Pi 5), las peticiones maliciosas externas provenientes de la IP atacante 178.139.228.237 quedan registradas de forma permanente en el archivo /var/log/apache2/access.log. El agente de Wazuh instalado en el host monitoriza este log en tiempo real y transmite los datos al Wazuh Manager (Ubuntu Server).

Ingesta y Configuración de Reglas Críticas (Wazuh XML)

La regla personalizada del SOC detecta firmas de exploit en las peticiones entrantes. Al coincidir los parámetros /api/deserialize o la firma rsc_payload en Base64, se dispara una alerta crítica mapeada con tácticas de MITRE ATT&CK:

/var/ossec/etc/rules/local_rules.xml 
<group name="web,app_moderna,">
  <!-- Regla base de monitorización HTTP de Apache -->
  <rule id="100500" level="3">
    <if_sid>31100</if_sid>
    <match>webinsegura1.duckdns.org</match>
    <description>Tráfico web dirigido al subdominio moderno del TFM</description>
  </rule>

  <!-- Regla crítica de detección del exploit React2Shell -->
  <rule id="100503" level="12">
    <if_sid>100500</if_sid>
    <match>/api/deserialize|rsc_payload|React2Shell</match>
    <description>SOC Critical Alert: Intento de explotación RCE React2Shell detectado en endpoint de deserialización</description>
    <mitre>
      <id>T1190</id> <!-- Exploit Public-Facing Application -->
      <id>T1203</id> <!-- Exploitation for Client Execution -->
    </mitre>

  </rule>

2. Playbook de Detección Desarrollado (Evidencia #19)

Este playbook de respuesta rápida se basa en las directrices de NIST SP 800-61 (Computer Security Incident Handling Guide) y el marco **SANS PICERL** para aplicaciones contenedorizadas. Dado que los atacantes utilizan redes dinámicas o IPs móviles (como la IP analizada 178.139.228.237) que rotan constantemente, el bloqueo de IPs individuales resulta ineficaz en escenarios reales. La contención se enfoca directamente en aislar la infraestructura de software afectada y aplicar un ciclo de actualización seguro (*Shift-Left Patching*).

Flujo de Mitigación en Aplicaciones Contenedorizadas

1. Alerta SIEM Alerta crítica RCE verificada desde IP 178.139.228.237
2. Contención Parada de Contenedor & Alerta Parar servicio. Notificar de inmediato al responsable técnico.
3. Erradicación Actualización & Pipeline Modificar código base, auditar con Semgrep y reconstruir imagen.
4. Recuperación Despliegue de Versión Segura Restaurar servicio y auditar salud en Wazuh.
Nivel Criterio de Evaluación Acción de Contención Correcta
NIVEL 1 Escaneo de vulnerabilidad con payloads HTTP de prueba que devuelven errores 400. Registrar IP dinámica y observar tendencias. No requiere bloqueo.
NIVEL 2 Múltiples peticiones POST exitosas pero sin comandos de consola confirmados. Escalar a L2 para inspección de memoria/procesos. Pre-alertar al administrador del sistema.
NIVEL 3 (Crítico) Ejecución de comandos confirmada en logs (RCE) mediante deserialización insegura. Contención Inmediata: Detener el contenedor del servicio afectado, notificar al responsable de la aplicación y proceder a la reconstrucción segura de la imagen.

3. Plan de Respuesta a Incidentes (PRI) (Evidencia #20)

A continuación se presenta la bitácora técnica de contención, erradicación y recuperación ante la detonación controlada del exploit React2Shell.

Bitácora Técnica de Eventos (Timeline Real del Laboratorio)

23:01:14 (Alerta SIEM / Identificación): El agente de Wazuh en el host detecta una petición POST anómala desde la IP externa real 178.139.228.237 al endpoint /api/deserialize con código de estado HTTP 200. Se dispara la alarma ID 100503.
23:02:30 (Investigación & Triage): El analista del SOC verifica los logs y constata una payload Base64 que contiene la firma del exploit React2Shell y la ejecución de los comandos whoami e id.
23:04:00 (Aislamiento & Contención): Se clasifica el incidente como crítico de nivel 3. Se procede a la detención física del contenedor Docker afectado para aislar el host y detener el compromiso de inmediato. Se emite un aviso urgente al responsable del servicio informando del incidente y del apagado controlado.
23:15:00 (Erradicación): El responsable técnico modifica el código base del backend, eliminando el uso de la función insegura eval() y aplicando validación estática Ajv sobre los parámetros de entrada. Se construye una nueva imagen segura.
23:25:00 (Recuperación y Cierre): Se despliega la versión corregida de la aplicación en producción. Se ejecutan pruebas automatizadas de validación del sistema completo para asegurar que no hay regresiones. Se confirma el retorno a la normalidad del dashboard de Wazuh.

1. Fase de Contención (Aislamiento de Servicio y Notificación)

Puesto que el atacante opera desde una IP móvil rotativa (178.139.228.237), aplicar reglas de bloqueo de red (como ufw deny) resulta insuficiente y poco profesional en producción. El procedimiento correcto consiste en detener la ejecución del microservicio vulnerado de inmediato para evitar movimientos laterales y exfiltración:

Comandos de terminal para aislamiento del microservicio 
# Detener preventivamente el contenedor comprometido en la Raspberry Pi 5
docker stop lab-modern-app

# Notificar mediante canal interno al responsable del servicio del incidente crítico
# [IR-Aviso-2304]: "Servicio 'lab-modern-app' detenido temporalmente tras alertar RCE desde IP 178.139.228.237."

2. Fase de Erradicación (Parche de Código Seguro y Actualización de Versión)

La erradicación exige realizar una actualización del código base para eliminar el parseo dinámico. Se inyecta la librería Ajv para compilar un JSON Schema seguro, neutralizando la deserialización de eval:

/app/api/deserialize/route.js - Solución Segura 
const Ajv = require("ajv");
const ajv = new Ajv();

// Definimos de forma restrictiva las propiedades permitidas para evitar inyección RCE
const componentSchema = {
  type: "object",
  properties: {
    name: { type: "string" },
    props: { type: "object" }
  },
  required: ["name"],
  additionalProperties: false
};

app.post('/api/deserialize', (req, res) => {
  try {
    const rawPayload = Buffer.from(req.body.rsc_payload, 'base64').toString();
    const parsedJSON = JSON.parse(rawPayload); // Evita la inyección dinámica (eval)

    const validate = ajv.compile(componentSchema);
    const valid = validate(parsedJSON);

    if (!valid) {
      return res.status(400).json({ error: "Payload no estructurado o inseguro" });
    }

    res.json({ status: "success", component: parsedJSON });
  } catch (err) {
    res.status(500).json({ error: "Error en procesamiento de componentes" });
  }
});
