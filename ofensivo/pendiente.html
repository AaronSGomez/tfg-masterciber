Arquitectura del Laboratorio

Es la primera imagen que examinará el tribunal de evaluación. Demuestra la topología del laboratorio y la separación lógica de componentes entre el atacante (Kali Linux), los servicios expuestos en la Raspberry Pi 5 y el SOC de monitoreo (Ubuntu Server con Wazuh).

📸 CAPTURA 1 — Diagrama de Arquitectura de Red y Laboratorio Recomendado: Draw.io / Excalidraw

¿Qué demuestra esta evidencia y por qué es importante?

Ilustra visualmente el flujo de comunicación y la distribución de red. Es fundamental para contextualizar cómo el atacante interactúa con la infraestructura.

Elementos que deben ser visibles en el diagrama:

  • Internet: Punto de origen/acceso público.
  • DuckDNS: Resolución dinámica de DNS (subdominios redirigidos a la IP pública del laboratorio).

  • Ubuntu Server: Instancia que aloja el servidor de Wazuh (SIEM/SOC).
  • Raspberry Pi 5: Dispositivo central que ejecuta los servicios vulnerables en Docker (Next.js vulnerable, DVWA, MariaDB, Nginx como proxy inverso).
  • Kali Linux: Nodo ofensivo del auditor/atacante.

HITO #5 — Acceso Web Explotado (SQL Injection en DVWA)

Interceptación de la petición HTTP POST de acceso.

📸 CAPTURA 7 — Login Exitoso con Bypass

Consecuencia inmediata del bypass por inyección SQL.

La cookie requerida para automatizar el ataque de SQLMap de forma autenticada.

Flujo de comandos ejecutados en Kali Linux:

Paso 1: Identificación e Inyección (Obtención de DBs)

Debe mostrar en consola: El listado completo de bases de datos disponibles (dvwa, mysql, information_schema).

[Captura 10: SQLMap output listing databases dvwa, mysql]

Plus: Herramientas Recomendadas para Destacar en el TFG

La adición de las siguientes herramientas de auditoría ofensiva le otorgará un carácter extremadamente técnico y profesional al informe final de tu TFG.

1. Reporte HTML Profesional de Nmap

Reconocimiento

Nmap puede exportar los resultados directamente a formato XML y traducirlos a HTML usando hojas de estilo XSLT integradas.

Ejecutar escaneo y conversión 
nmap -sCV -O -oA reconocimiento webinsegura1.duckdns.org
xsltproc reconocimiento.xml -o reconocimiento.html

📸 Captura extra sugerida: Abre `reconocimiento.html` en el navegador. Mostrará una tabla interactiva muy profesional con servicios, puertos y huella del SO.

2. nuclei (Escaneo Automatizado de CVEs)

Exploración / CVEs

Herramienta moderna de escaneo de vulnerabilidades basada en plantillas YAML que permite validar CVEs específicos de forma directa.

Lanzar escaneo de Nuclei 
nuclei -u https://webinsegura1.duckdns.org

📸 Captura extra sugerida: Reporte en terminal con la detección de exploits específicos aplicados a la URL.

3. WhatWeb (Identificación de Tecnologías)

Reconocimiento

Identifica y clasifica de forma rápida las tecnologías subyacentes del servidor web.

Ejecutar WhatWeb 
whatweb https://webinsegura1.duckdns.org

Debe verse en consola: NextJS, React, Nginx, TLS, IP del servidor, etc.

4. testssl.sh (Análisis Avanzado SSL/TLS)

Cifrado / Endurecimiento

Herramienta por consola para probar el puerto de cifrado en cualquier dirección IP o dominio remoto.

Ejecutar testssl 
testssl.sh webinsegura1.duckdns.org

Debe reportar en salida: Habilitación de TLS 1.3, Cipher Suites vigentes, cabecera HSTS configurada y Perfect Forward Secrecy.

5. CVSS Calculator (Calculadora Oficial FIRST)

Mapeo de Severidad

Presenta la matriz matemática oficial utilizada por los auditores internacionales para cuantificar el riesgo de los hallazgos.

Acceso: Calculadora CVSS v3.1 Oficial de FIRST

HITO #8 — Estructura del Informe Final de Pentest y Soluciones

Estructuración formal y académica del reporte técnico. Como estas máquinas se mantienen vulnerables de forma intencionada para fines demostrativos en el laboratorio, se documenta a continuación la remediación oficial requerida para parchear dichos fallos en entornos de producción.

REMEDIACIÓN

Solución a React2Shell (RCE)

La vulnerabilidad de inyección/deserialización de comandos (CVE-2025-55182) reside en el protocolo de componentes de servidor de Next.js.

Acción Correctiva Oficial:

  • Actualización de Framework: Actualizar Next.js en el package.json a la versión 15.5.7 o superior, donde se restringe y sanitiza el payload de des-serialización RSC.
  • Aislamiento de Ejecución: Asegurar que los contenedores Node no ejecuten la aplicación con privilegios root (usar directiva USER node en Dockerfile).
REMEDIACIÓN

Solución a SQL Injection (DVWA)

El bypass de autenticación y volcado de bases de datos se produce por la concatenación de variables en la consulta SQL directamente.

Acción Correctiva Oficial:

  • Consultas Preparadas (Prepared Statements): Reemplazar consultas inseguras por sentencias preparadas de PDO o sentencias parametrizadas nativas en PHP/Node para evitar la inyección de código como datos.
  • Principio de Mínimo Privilegio: Configurar el usuario MariaDB para que carezca de acceso de lectura fuera de la base de datos de producción necesaria.
