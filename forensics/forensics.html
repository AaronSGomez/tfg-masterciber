Manual Forense: Adquisición Lógica y Análisis con Autopsy

Guía de análisis forense digital bajo directrices **NIST SP 800-86**: desde la extracción física y lógica del volumen comprometido de base de datos MariaDB (Raw Image `dd`), la verificación criptográfica SHA-256, hasta la investigación minuciosa en la plataforma **Autopsy**.

1. Principios de Integridad y Cadena de Custodia

El análisis forense digital se rige por la preservación rigurosa de las pruebas. La regla fundamental es nunca realizar análisis directamente sobre el dispositivo o volumen de almacenamiento original. Se debe realizar una copia bit a bit (clonado físico o adquisición lógica) y validar su integridad matemática utilizando funciones hash criptográficas (como MD5 o SHA-256).

Preservación

Aislamiento del volumen de almacenamiento comprometido antes de realizar cualquier lectura para impedir que el sistema de archivos altere metadatos críticos como las fechas MAC (Modify, Access, Create).

Verificación

Generación de la firma hash SHA-256 del dispositivo origen y su posterior contraste con el archivo de imagen generado (`.dd`). Ambos hashes deben coincidir con precisión absoluta.

Documentación

Registro minucioso de cada fase de adquisición, incluyendo marcas temporales, comandos ejecutados, herramientas utilizadas y firmas del analista forense a cargo.

2. Adquisición del Volumen Comprometido

En el escenario del TFM, la base de datos MariaDB (que contenía las tablas y hashes vulnerados en la fase ofensiva) se aloja dentro del contenedor lab-db-server con un volumen mapeado persistente. El proceso de adquisición se centra en realizar un volcado *raw* (copia exacta de bloques) de dicho volumen de datos persistente.

Comandos de Adquisición en Consola Forense (Linux)

Paso a paso técnico para la localización, firma y volcado del volumen comprometido de MariaDB:

Bash Forensic Work - Adquisición dd 
# 1. Identificar la ruta del volumen de datos del contenedor Docker de MariaDB
docker volume inspect ciber_db_data

# 2. Generar el hash SHA-256 del bloque/directorio origen antes de la lectura
sudo sha256sum /var/lib/docker/volumes/ciber_db_data/_data/ibdata1

# 3. Realizar la copia exacta bit a bit en formato raw (.dd / .raw) del archivo ibdata1
sudo dd if=/var/lib/docker/volumes/ciber_db_data/_data/ibdata1 of=/forensics/mariadb_ibdata1.raw bs=4M status=progress

# 4. Generar y registrar la firma hash del archivo de salida para verificar su integridad
sha256sum /forensics/mariadb_ibdata1.raw

# 5. Comparar ambos hashes. El hash del archivo de salida debe coincidir perfectamente:
# [Consola]: f896207cda7e271a3378fb8f731110022f30ee941199341aa5019889ff0277bd /forensics/mariadb_ibdata1.raw

3. Manual de Ejecución Paso a Paso en Autopsy

Sigue esta guía técnica estructurada para configurar, cargar y analizar la imagen forense obtenida en el laboratorio:

1 Creación de Caso (Case Setup)

1. Inicia Autopsy y haz clic en **"New Case"**.
2. En la ventana emergente, introduce el nombre del caso: MariaDB_Security_Breach_Incident.
3. Elige la **Base Directory** de tu equipo para alojar los datos de indexación del caso.
4. Configura los metadatos del investigador:

  • Case Number: TFM-SOC-IR-2026
  • Examiner Name: Aarón S. Gómez
  • Contact Details: (Email y teléfono para firmar el reporte final).

2 Carga e Ingesta del Origen de Datos (Add Data Source)

1. Selecciona **"Add Data Source"** en la barra superior.
2. Marca el tipo de fuente como **"Disk Image or VM File"** y pulsa Siguiente.
3. En la ruta del archivo, selecciona la imagen extraída: mariadb_ibdata1.raw.
4. Verificación de Integridad: En el cuadro **"Hash Values"**, introduce el hash SHA-256 obtenido en la fase de adquisición (f896207cda7e271a3378fb8f731110022f30ee941199341aa5019889ff0277bd). Autopsy verificará que el archivo coincida bit a bit antes de indexarlo.

3 Configuración de Módulos de Ingesta (Ingest Modules)

Para optimizar el tiempo de ejecución y centrar la telemetría en la base de datos, marca/desmarca los módulos siguiendo estas pautas:

  • File Type Identification: Activo. Compara los números mágicos del archivo para validar que ibdata1 sea en efecto una estructura InnoDB válida.
  • Extension Mismatch Detector: Activo. Alertará si se ha renombrado algún fichero de base de datos o logs para esconderlo.
  • Keyword Search (Configuración Manual): Activo. Haz clic en este módulo y ve a **"Manage Lists"**. Crea una lista nueva llamada TFM_Exfiltracion_Firma y añade los siguientes términos clave a indexar en crudo:
    - db_modern_user
    - InsecureAIPassword_123
    - flag.txt
    - AARONSGOMEZ
  • Recent Activity / Email Parser / Central Repository: Desactiva estos módulos. Al ser una imagen lógica del motor InnoDB (`ibdata1`) y no una imagen completa del sistema operativo Windows/Linux, estos módulos ralentizan la ingesta de manera innecesaria.

4. Metodología de Análisis Forense de Base de Datos

Una vez completada la ingesta en Autopsy, la estructura interna del volumen InnoDB se indexa en memoria. Sigue este procedimiento para extraer las evidencias del compromiso del servidor de base de datos:

A. Inspección en el Visor Hexadecimal

Haz clic derecho sobre el archivo mariadb_ibdata1.raw y selecciona **"View in Hex"**. InnoDB organiza la base de datos en páginas de 16KB. Las páginas de índice almacenan las tuplas de datos de forma secuencial.

Desplázate por el visor para buscar los offsets y las cabeceras binarias típicas de MariaDB. Los registros que han sido borrados de las tablas permanecen visibles en crudo en el espacio libre (*Slack Space*) hasta que se realice una operación OPTIMIZE TABLE o nuevas transacciones sobrescriban esos bloques físicos.

B. Ejecución de la Lista de Palabras Clave

Haz clic en el nodo **"Keyword Hits"** en el árbol lateral izquierdo de Autopsy. Selecciona tu lista TFM_Exfiltracion_Firma.

La consola listará las páginas exactas de InnoDB donde se han localizado las cadenas. Por ejemplo, al buscar InsecureAIPassword_123, Autopsy revelará la página de datos que contiene las contraseñas de las cuentas de usuario exfiltradas por inyección SQL, proporcionando la evidencia forense irrefutable del compromiso de la integridad del almacenamiento del TFM.
