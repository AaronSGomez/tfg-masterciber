Guía Paso a Paso: Búnker de Seguridad con Nginx y Docker

Documentación de la arquitectura de defensa perimetral para aislar un backend del acceso público. Se aplican políticas de restricción máxima (Zero Trust y Mínimo Privilegio) limitando drásticamente el tamaño de las peticiones, mitigando ataques DDoS y auditando el tráfico mediante un proxy inverso Nginx contenedorizado.

Entendiendo la Arquitectura de Aislamiento

La seguridad real empieza por la arquitectura física de la red. En lugar de exponer la base de datos o la aplicación directamente a internet, creamos capas de protección sucesivas (**Arquitectura de Defensa en Profundidad**).

Concepto Clave 🛡️

El Backend y la Base de Datos se ejecutan en una red privada virtual de Docker (secure_net). No tienen puertos expuestos al servidor físico host. El único que puede comunicarse con ellos es Nginx, y Nginx solo acepta peticiones que vengan del proxy principal (Apache) a nivel local.

Esquema Lógico del Tráfico (Zero-Trust)
Público / Internet HTTPS (Puerto 443)
Apache (Host) Redirecciona localmente
Nginx Container (Proxy) Escucha en 127.0.0.1:8081 Rate limit & Max 56k
Red Privada
Backend & DB Sin puertos abiertos al host

Aislamiento Físico (Docker Compose)

El primer paso es configurar nuestro entorno de contenedores para cortar el acceso directo al backend y establecer a Nginx como el único embudo de entrada.

docker-compose.yml 
services:
  api_backend:
    build: .
    container_name: app_backend
    restart: always
    # ELIMINADO: El bloque 'ports' se suprime para hacerlo invisible al exterior
    networks:
      - secure_net

  proxy_nginx:
    image: nginx:alpine
    container_name: app_nginx
    ports:
      # RESTRICCIÓN MÁXIMA: Nginx expone su puerto 80 SOLO al localhost (127.0.0.1)
      - "127.0.0.1:8081:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
    depends_on:
      - api_backend
    networks:
      - secure_net

networks:
  secure_net:
    driver: bridge
Justificación de Seguridad 🛡️

Al atar Nginx estrictamente a 127.0.0.1:8081, bloqueamos por diseño que un atacante intente saltarse Apache escaneando o atacando la IP pública del servidor por el puerto 8081. Nginx será completamente sordo a peticiones externas no enrutadas.

Cortafuegos Restrictivo en Nginx

Asumiendo que todo tráfico entrante es potencialmente hostil, se configuran reglas en Nginx con umbrales extremadamente bajos, ajustados milimétricamente al uso real de la API.

nginx.conf (Bloque http) 
http {
    # 1. Ocultar huella del servidor a los atacantes (Evita perfilado de CVEs)
    server_tokens off;

    # 2. Zona de memoria para control estricto de IP (10MB) a un ritmo de 5 peticiones/seg
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=5r/s;

    # 3. Balanceador de Carga: Distribuye el tráfico hacia las réplicas del Backend
    upstream backend_servers {
        server api_backend:8080; 
        # Aquí se podrían añadir más réplicas físicas del contenedor backend
        # server api_backend_replica2:8080;
    }
    # ...
}

Reglas de Protección (Bloque server)

Dentro del servidor Nginx, aplicamos barreras diseñadas para estrangular cargas maliciosas antes de que consuman ciclos de CPU o memoria en el Backend.

Las peticiones JSON estándar envían en promedio unos 3kb; con un tope de 56k se tiene un margen de seguridad sumamente amplio para cualquier petición legítima, pero se bloquea de cuajo cualquier intento de subida de archivos o inyección de payloads gigantes.

nginx.conf (Bloque server) 
server {
    listen 80;

    # BARRERA 1: Bloqueo de Payloads Pesados (Restricción a 56 Kilobytes)
    client_max_body_size 56k;

    # BARRERA 2: Timeouts Anti-Zombies (Corte agresivo a 10s)
    client_body_timeout 10s;
    client_header_timeout 10s;
    send_timeout 10s;

    location / {
        # BARRERA 3: Ejecución implacable del Rate Limiting
        limit_req zone=api_limit burst=10 nodelay;

        proxy_pass http://backend_servers;
    }
}
Límite Asfixiante de 56k

Calculado basándose en el peso real máximo de un JSON legítimo. Bloquea instantáneamente (Error 413) intentos de desbordamiento de búfer o subida de binarios.

Timeouts Agresivos (10s)

Mitiga ataques tipo "Slowloris", cortando conexiones de clientes que envían cabeceras a cuentagotas para agotar los hilos disponibles en el servidor.

Burst=10 nodelay

Permite una ventana inicial de 10 peticiones rápidas para cargas iniciales, pero bloquea sin piedad (Error 503) ráfagas de fuerza bruta o escaneos rápidos.

Auditoría y Trazabilidad (Logs)

Dado que la API opera detrás de múltiples proxies (Apache → Nginx → Docker), las IPs entrantes podrían enmascararse como locales. Se implementa una trazabilidad absoluta para registrar el origen real del tráfico.

nginx.conf (Cabeceras y Logs) 
# 1. Formato personalizado para capturar la IP real y el Vector de Ataque
log_format main '$remote_addr - $remote_user [$time_local] "$request" '
                '$status $body_bytes_sent "$http_referer" '
                '"$http_user_agent" "$http_x_forwarded_for"';

# 2. Aplicar la política de logs
access_log /var/log/nginx/access.log main;

location / {
    # 3. Inyectar la IP real a las cabeceras hacia el Backend
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_pass http://backend_servers;
}

Comandos de Inspección (Respuesta a Incidentes)

Comandos operativos esenciales para monitorizar el tráfico en tiempo real, cazar anomalías y detectar intentos de intrusión:

Comandos de Auditoría en Tiempo Real 
# Monitorizar el tráfico en tiempo real atravesando las defensas
docker logs -f app_nginx

# Cazar IPs bloqueadas por el limitador de velocidad (Rate Limiting)
docker logs app_nginx | grep "limiting requests"

# Identificar intentos de inyección de payloads gigantes (>56KB)
docker logs app_nginx | grep "client intended to send too large body"

Pruebas de Carga y Bloqueo en Tiempo Real (OWASP ZAP & Dozzle)

Balanceador de Carga y Proxy Inverso

Nginx no solo opera como un cortafuegos perimetral, sino que actúa como un **Balanceador de Carga** robusto en la capa de aplicación. Mediante el bloque upstream backend_servers, el proxy inverso es capaz de repartir de forma inteligente las peticiones entrantes entre distintas réplicas del contenedor de la API (Scale-Out), maximizando la alta disponibilidad del servicio y evitando cuellos de botella.

Detonación de Ataque con OWASP ZAP

Durante el transcurso de las sesiones prácticas, realizamos una simulación real de ataque ejecutando un **escaneo dinámico invasivo mediante OWASP ZAP** contra la ruta expuesta de la aplicación.

Al lanzar cientos de peticiones automatizadas consecutivas por segundo con payloads inyectables, **Nginx entró en acción de forma automática**:

  • Activó la zona de memoria api_limit al superarse la tasa de 5 peticiones/segundo.
  • Descartó de forma fulminante el tráfico excedente del escáner ZAP devolviendo un error **HTTP 503 (Service Temporarily Unavailable)**.
  • Bloqueó la inyección de payloads de fuzzing pesados al superar el límite estricto de **56KB**, respondiendo de forma inmediata con un error **HTTP 413 (Payload Too Large)**.

Visualización en Tiempo Real con Dozzle

Para supervisar la respuesta activa del cortafuegos sin necesidad de ejecutar comandos por consola en la Raspberry Pi 5, desplegamos **Dozzle**, un visor web de logs de Docker en tiempo real, liviano y auto-hospedado.

Despliegue de Dozzle (docker-compose.yml) 
services:
  dozzle:
    container_name: log_viewer_dozzle
    image: amir20/dozzle:latest
    ports:
      # Expone la interfaz web del lector de logs en el puerto 8888
      - "127.0.0.1:8888:8080"
    volumes:
      # Vinculación del socket del demonio de Docker para lectura en vivo
      - /var/run/docker.sock:/var/run/docker.sock:ro
    restart: always

Efecto del Bloqueo 🛑

Evidencia Operativa: Las trazas demuestran cómo la IP atacante 192.168.1.85 (Kali Linux con ZAP) fue interceptada de inmediato por las directivas de seguridad. El primer log muestra el estrangulamiento de velocidad (Rate Limiting), y el segundo log muestra el bloqueo instantáneo del payload que superó la barrera de 56K configurada en Nginx.

Caso de Estudio: Detección de Cuentas Troll en Damtrivia y Transición a Google OAuth

Además del filtrado perimetral en Nginx, la auditoría y centralización de logs nos permite identificar comportamientos anómalos o abusos a nivel de lógica de negocio. Durante las pruebas de la aplicación de test Damtrivia, la supervisión de logs del backend (Spring Boot) delató un ataque de registro automatizado de cuentas troll con direcciones de correo ficticias u ofensivas que evadían las políticas básicas del sistema.

Resolución mediante Hardening 🛡️

Acción Correctiva: Tras identificar la IP pública atacante 95.127.144.50 y la persistencia de registros ofensivos directos, se implementó la **autenticación OAuth 2.0 de Google**. Esta delegación de identidad obliga a los usuarios a autenticarse mediante una cuenta Google verificada y real para registrarse, eliminando por completo el anonimato de creación de cuentas troll. Esto resuelve un problema que el bloqueo clásico de IP ya no soluciona por sí solo, dado que hoy en día cualquier atacante dispone de un acceso sumamente sencillo a IPs móviles y dinámicas que cambian constantemente. Este nuevo método facilita el bloqueo y control de trolls independientemente de la dirección IP de origen empleada para el trolleo.
